### T3.12 — Sass `@import` → `@use` migration (top-level partials only)

**Scope cap:** Migrate only the top-level partials referenced from `assets/css/main.scss`. Font-awesome's internal `@import` chain stays untouched — it's a vendored library and would require either upgrading to a `@use`-compatible version or refactoring the included files. Documented in the commit + next-set queue.

**Migration plan:**

1. **`_sass/_variables.scss`:** Add `!default` to `$max-content-width` definition near the top:
   ```scss
   $max-content-width: 930px !default;
   ```
   This is new — currently `$max-content-width` is only defined in main.scss with a Liquid template value. The `!default` lets main.scss override it.

2. **Each partial that references variables:** Add `@use 'variables' as *;` at the top.
   - `_base.scss`: needs `$black-color`, `$white-color`
   - `_layout.scss`: needs `$max-content-width`
   - `_fonts.scss`: needs font path vars
   - `_geist.scss`: needs `$geist-font-path`
   - `_themes.scss`: needs many vars

3. **Each independent partial:** No prepended directive needed. `@use` from main.scss with no `as` clause is enough.
   - `_cv.scss`, `_distill.scss`, `_dropdown.scss`, `_tabs.scss`, `_typograms.scss`

4. **`assets/css/main.scss`:** Replace `@import` block with `@use`:
   ```scss
   ---
   ---
   @charset "utf-8";

   // Configure $max-content-width (Liquid-templated from _config.yml).
   @use "variables" with (
     $max-content-width: {{ site.max_width }}
   );
   @use "geist";
   @use "fonts";
   @use "themes";
   @use "layout";
   @use "base";
   @use "distill";
   @use "cv";
   @use "tabs";
   @use "typograms";
   @use "dropdown";

   // Vendored libraries — still use @import internally (separate migration).
   @import
     "font-awesome/fontawesome",
     "font-awesome/brands",
     "font-awesome/solid",
     "font-awesome/regular",
     "tabler-icons/tabler-icons.scss",
     "tabler-icons/tabler-icons-filled.scss",
     "tabler-icons/tabler-icons-outline.scss"
   ;
   ```

5. **Test the build.** Expect a smaller number of deprecation warnings (only font-awesome's internal warnings remaining; the 11 top-level main.scss warnings should be gone).

6. **If specific cross-partial dependencies appear** (e.g., `_themes.scss` needing something from `_base.scss`), add explicit `@use` statements as needed.

**Verification:**
- Jekyll build completes (no Sass errors).
- Diff `_site/assets/css/main.css` before vs after migration — visual diff on key selectors should be empty (CSS output should be effectively identical).
- All five tested pages render visually identical: `/`, `/publications/`, `/talks/`, `/services/`, `/football/`. Both light and dark themes.
- Mobile 375px renders correctly.
- Sass deprecation warning count drops noticeably in the build log.

**Risk:** highest of the bundle. Possible failure modes:
- Variable scoping bug — a partial references a variable it didn't `@use`. Caught by build error or empty CSS rule. Fix by adding the `@use`.
- `with` configuration syntax error. Caught by build error.
- Output CSS differs subtly (missing rule, wrong order). Caught by visual comparison; bisected by reverting individual partial migrations.

**Rollback:** the migration is a single commit (or two — `_sass/` + `main.scss`). If post-deploy verification finds visual regressions, revert that commit. Other items in this session are independent and stay in.
